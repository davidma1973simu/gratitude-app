# 感恩日记 v2 开发计划

## 核心理念

**"留住这束光"** — 让用户在给予感恩的瞬间，意外地收到过去的回响。

情感触碰的前提不是"看到旧内容"，而是"在对的心境下看到"。
用户最柔软的时刻是**刚写完感恩、点击提交后**。
此时让他"收到一条过去的感恩"，不是打扰，是回响。

---

## v2 三个新功能

### 功能 1：提交后的回响（Post-Submission Echo）

**触发时机**：用户提交今天的感恩后，感谢弹窗底部自然延展一条过去的记录。

**交互流程**：
1. 用户写完三条 → 点提交 → 弹出"感谢你记录今天的光"
2. 弹窗底部，0.8秒后淡入一条旧感恩（不弹、不跳、不打断）
3. 文案示例：

```
─────────────────────
"那个下雨天，有人递了一把伞"
            ── 187天前
─────────────────────
```

4. 用户可以直接关掉弹窗，也可以点那条旧感恩 → 进入"重逢"全屏

**技术要点**：
- 提交成功后从本地缓存中筛选一条旧记录
- 感谢弹窗底部追加 DOM，CSS 动画淡入
- 不改变现有提交流程，纯增量

---

### 功能 2：首页水印光（Homepage Watermark）

**触发时机**：用户打开 APP，问候语下方，一条旧感恩以极淡方式呈现。

**交互流程**：
1. 首页问候语（"晚上好 🌙"）下方
2. 一行极淡的文字，opacity 约 0.3-0.4，几乎是背景的一部分
3. 不加边框、不加卡片样式，就是安静的一行字
4. 配日期标注（"187天前" 或 "去年今天"）
5. 点击 → 进入"重逢"全屏
6. **用户可以直接无视，开始写今天的感恩——零干扰**

**视觉示例**：
```
晚上好 🌙

今天的你，有什么值得感恩的？

        "那个下雨天，有人递了一把伞"
                    ── 187天前

[输入框1]
[输入框2]
[输入框3]
```

**技术要点**：
- `startApp()` 渲染首页时，追加水印 DOM
- 低 opacity + 较小字号 + 无交互样式 = 视觉退让
- hover 时微微变亮（opacity 0.6），暗示可点击

---

### 功能 3：重逢全屏 + 留住这束光（Reunion View + Image Card）

**触发时机**：用户点击水印光或弹窗中的旧感恩 → 展开沉浸式全屏。

**交互流程**：
1. 全屏暗色/暖色背景（不是白底，是深棕/深墨绿）
2. 日期大字居中（"2025年12月3日" 或 "187天前"）
3. 三条感恩内容逐条淡入（间隔 0.5s）
4. 底部一行小字：*"你曾为这些瞬间心动"*
5. 右下角一个极简的 **"留住这束光"** 按钮
6. 点击 → 生成图片卡片（Canvas 导出 PNG）

**图片卡片设计**：
- 竖版，适合手机屏幕和社交分享
- 深色背景 + 暖光效果
- 日期 + 三条感恩文字 + 🌿 图标
- 底部极小字：感恩日记
- 用 Canvas API 纯前端生成，无需后端

**技术要点**：
- 新增 `#reunionView` 全屏覆盖层
- CSS 动画：淡入、逐条出现
- Canvas API 绘制图片卡片 → `toDataURL()` → 下载/分享
- 分享用 `navigator.share()`（支持时）+ 降级为下载

## 旧感恩选择算法

优先级（逐级降级，命中即返回）：

1. **同日周年** — 去年今天、前年今天。巧合本身就有力量
2. **100天前** — 不近不远，刚好遗忘又被想起
3. **30-365天随机** — 加权随机，越远权重略高（惊喜感更强）
4. **静默** — 记录少于7条时不显示，宁缺毋滥

算法函数签名：
```js
function pickMemory(entries, today) → { entry, label }
// entry: 旧感恩记录对象
// label: "去年今天" | "187天前" | "312天前"
// returns null if no suitable entry
```

## A/B 测试方案

### 方案：同域双路径

| 版本 | URL | 说明 |
|------|-----|------|
| v1（当前） | `/gratitude-app/` | 保持不变，继续维护 |
| v2（新版） | `/gratitude-app/v2/` | 新增目录，独立页面 |

**为什么不用分支/子域**：
- 同域部署，共享同一个 GitHub Pages 和 Service Worker 基础
- 用户只需要一个不同链接就能访问不同版本
- 数据都进同一个 Supabase（同一套 entries 表），因为用户数据是通用的
- 分发时可以给不同人发不同链接

**文件结构**：
```
gratitude-app/
├── index.html          ← v1，不动
├── v2/
│   └── index.html      ← v2，全新文件，包含回响+水印+重逢
├── manifest.json       ← v1
├── v2/manifest.json    ← v2（如需不同的 PWA 配置）
├── sw.js               ← 共用，需更新缓存策略
├── icon-192.png
└── icon-512.png
```

### A/B 测试执行

1. 给 A 组用户发 `davidma1973simu.github.io/gratitude-app/`（v1）
2. 给 B 组用户发 `davidma1973simu.github.io/gratitude-app/v2/`（v2）
3. 观察 1-2 周：
   - v2 用户是否更频繁打开（留存率）
   - v2 用户"留住这束光"功能使用率
   - 主观反馈（问卷或访谈）

### 数据埋点（可选，轻量）

在 Supabase 新建 `events` 表记录关键行为：
```sql
CREATE TABLE events (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  user_id UUID REFERENCES auth.users(id),
  event TEXT NOT NULL,        -- 'submit', 'reunion_view', 'keep_light', 'share'
  version TEXT NOT NULL,      -- 'v1' or 'v2'
  meta JSONB,                 -- 额外信息，如天数差
  created_at TIMESTAMPTZ DEFAULT now()
);
```

## 不做的事

- ❌ 连续打卡/勋章/积分 — 这不是打卡产品
- ❌ 推送通知提醒回溯 — 打扰用户
- ❌ 社交功能（关注/点赞）— 不是社交产品
- ❌ 数据分析仪表盘 — 情感产品不需要 KPI
- ❌ AI 生成感恩内容 — 这是用户的真实记忆，不是 AI 的作业
