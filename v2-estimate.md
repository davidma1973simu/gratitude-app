# 感恩日记 v2 开发评估

## 1. 时间预估

| Phase | 任务 | 预估时间 | 说明 |
|-------|------|---------|------|
| Phase 0 | A/B 架构准备：创建 v2 目录、复制 v1 为基础、调整路径 | 15 分钟 | 纯文件操作 |
| Phase 1 | `pickMemory` 选择算法：同日周年、100天前、加权随机、静默条件 | 20 分钟 | 算法逻辑清晰，代码量少 |
| Phase 2 | 提交后回响：修改感谢弹窗，追加旧感恩 DOM + CSS 淡入 | 30 分钟 | 需仔细调整弹窗布局和动画时机 |
| Phase 3 | 首页水印光：问候语下方低 opacity 文字 + hover 微交互 | 20 分钟 | 纯 DOM/CSS 增量，改动最小 |
| Phase 4 | 重逢全屏：`#reunionView` 暗色背景 + 逐条淡入 + i18n | 45 分钟 | 新视图，CSS 动画 + 双语文案较多 |
| Phase 5 | 留住这束光：Canvas 绘制图片卡片 + 下载/分享 | 60 分钟 | **最复杂**——Canvas 绘制需要精确计算文字排版、渐变背景、自动换行 |
| Phase 6 | 部署 & 测试：推送 gh-pages、两端分别验证 | 15 分钟 | 标准操作 |

**纯开发时间：约 3 小时**
**含调试和迭代：约 4-6 小时（实际完成时间）**

最可能卡住的环节：
- Phase 5 Canvas 图片卡片：文字自动换行、中英文混排、字体大小适配不同屏幕
- Phase 4 重逢视图的动画节奏感需要反复微调才能达到"情感触碰"的效果

---

## 2. 任务复杂度

| 维度 | 评级 | 说明 |
|------|------|------|
| 代码量 | ⭐⭐ 中等 | v1 是 1567 行 / 64KB；v2 预估增加 ~400-500 行（pickMemory + 水印DOM + reunionView + Canvas） |
| 技术难度 | ⭐⭐⭐ 中偏高 | Canvas API 绘制是唯一的技术挑战点，其余都是 DOM/CSS 增量 |
| 设计难度 | ⭐⭐⭐ 中偏高 | "情感触碰"的视觉节奏需要精细调校——淡入速度、opacity 层级、暗色配色 |
| 逻辑复杂度 | ⭐⭐ 低 | pickMemory 算法简单，回响和水印是纯 UI 增量，不影响核心存储逻辑 |
| i18n 扩展 | ⭐⭐ 中等 | v2 新增文案约 15-20 条，需要同时写中英两种 |

**整体复杂度：中等偏上**
——不是因为逻辑难，而是因为"让用户被触动"这件事需要在视觉和节奏上反复打磨。

## 3. 外部资源需求

| 资源 | 是否需要 | 说明 |
|------|---------|------|
| Supabase | ✅ 已有 | v2 和 v1 共用同一个 Supabase 实例，无需新建。entries 表结构不变 |
| Supabase events 表 | ⭐ 可选 | 如果要做 A/B 数据埋点，需新建 `events` 表。免费 tier 完全覆盖 |
| 域名 | ⭐ 可选 | ganen.app / 3thanks.app（¥70-100/年/个），提升分发信任感，但不影响开发 |
| CDN/图片服务 | ❌ 不需要 | 图片卡片纯前端 Canvas 生成，不依赖任何外部存储 |
| 第三方 JS 库 | ❌ 不需要 | 继续零依赖策略，Canvas API 是浏览器原生 |
| 邮件服务 | ❌ 不需要 | 登录用邮箱+密码，不走邮件 |
| 支付/变现 API | ❌ 不需要 | v2 不涉及商业化 |

**关键结论：v2 开发不需要任何新的外部资源或付费服务。**

唯一可选支出是域名（与 v1 共用问题，独立于开发）。

## 4. AI Credit 预估

| Phase | 预估 AI turns | 说明 |
|-------|---------------|------|
| Phase 0 | 3-5 | 文件操作，简单 |
| Phase 1 | 5-8 | 算法实现 + 测试 |
| Phase 2 | 8-12 | 弹窗修改需要反复调整样式 |
| Phase 3 | 5-8 | DOM/CSS 增量 |
| Phase 4 | 10-15 | 新视图 + 动画 + i18n，需要打磨 |
| Phase 5 | 15-20 | Canvas 绘制最耗时——文字排版、渐变、换行、中英混排、多设备适配 |
| Phase 6 | 3-5 | 部署操作 |
| **总计** | **50-70 turns** | |

**对应 WorkBuddy Credit 消耗**：
- 每个 turn 平均消耗约 3,000-8,000 tokens（视改动大小）
- 总 token 消耗预估：200K-500K tokens
- 对应 Credit：约 **30-60 Credit**（按标准计费模型估算）

**Credit 节省策略**：
- Phase 1-3 可以一次性完成（逻辑简单、改动小），减少来回对话
- Phase 5 可以先做最小可用版（纯文字卡片），再迭代美化，避免一次性追求完美
- 每次提交代码前一次性改完所有相关部分，不要零碎修改

## 5. 风险和注意事项

| 风险 | 级别 | 对策 |
|------|------|------|
| Canvas 图片卡片中英文排版错乱 | 中 | 先用固定字号测试中文，再用固定字号测试英文，最后做自适应 |
| 水印光打扰用户输入 | 低 | opacity 控制在 0.25-0.35，hover 才变亮，实测调整 |
| 重逢动画节奏不打动人 | 中 | 需要真人体验反馈，预计需要 2-3 次迭代 |
| v1 和 v2 SW 缓存冲突 | 低 | v2 使用独立缓存 key（`gratitude-v2-v1`） |
| Supabase RLS 对 v2 有影响 | 无 | v2 和 v1 共用同一个 entries 表和 RLS 策略，零影响 |

---

## 6. 推荐执行策略

**建议分两次 session 完成：**

**Session 1（约 2 小时）**：Phase 0 → 1 → 2 → 3
- 搭建 v2 基础，实现算法、回响和水印
- 这三个 Phase 改动小、逻辑清晰，可以一气呵成
- 完成后部署测试，确认基础功能正常

**Session 2（约 2-3 小时）**：Phase 4 → 5 → 6
- 重逢全屏 + 留住这束光
- 这两个 Phase 是 v2 的灵魂，需要更多打磨时间
- 完成后部署，进入 A/B 测试阶段

**两次 session 之间可以先用 v2 beta 测试回响和水印效果，收集反馈再决定重逢的视觉细节。**
